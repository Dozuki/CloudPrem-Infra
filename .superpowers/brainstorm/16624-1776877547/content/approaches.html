Blue/Green Deployment: Three Approaches

All approaches use Aurora MySQL Blue/Green Deployments for zero-data-loss DB switchover. The difference is how compute (EKS + app) is handled.

A

Single Cluster, Dual Namespace (Recommended)

One EKS cluster with two namespaces: dozuki-blue and dozuki-green. Each namespace gets its own full app deployment. Envoy Gateway routes traffic between them via weighted HTTPRoutes.

Pros

  • Lowest cost — one cluster, shared nodes
  • Fastest spin-up — just deploy a new namespace
  • Envoy Gateway already supports weighted routing
  • Karpenter auto-provisions nodes for green
  • Works with existing Spacelift physical/logical split

Cons

  • No infra-level isolation (shared cluster)
  • Can't test EKS upgrades in isolation
  • Noisy neighbor risk between blue/green
  • Still needs Aurora B/G for database layer
B

Dual Cluster, Shared Data Layer

Two EKS clusters in the same VPC, sharing RDS (Aurora), ElastiCache, S3. NLB target groups switch between cluster A and cluster B. Full infra isolation for compute.

Pros

  • Full compute isolation — test EKS upgrades safely
  • Can validate infra changes on green cluster
  • Clean blast radius — green failure can't affect blue
  • NLB target group switching for traffic control
  • Aurora B/G handles database switchover natively

Cons

  • Higher cost — two EKS clusters (spin up on deploy)
  • Slower spin-up — EKS cluster creation ~15 min
  • More complex Terraform (two physical stacks)
  • Cross-cluster service discovery needed
C

Dual Stack (Full Isolation)

Two complete stacks — each with its own EKS cluster, Aurora instance (with B/G replication between them), ElastiCache, and NLB. Route53 weighted routing shifts traffic between stacks.

Pros

  • Complete isolation — nothing shared
  • Can test everything including network changes
  • Route53 weighted routing is battle-tested
  • Cleanest rollback — just flip DNS weights
  • Each stack is independently deployable

Cons

  • Highest cost — double everything
  • Slowest spin-up — full stack creation ~30+ min
  • Most complex orchestration
  • S3 bucket sharing/sync needed
  • ElastiCache cold start on green

My Recommendation: Approach B (Dual Cluster, Shared Data)

You specifically asked for infra-level blue/green (EKS upgrades, Terraform changes). Approach A only covers app-level. Approach C is overkill — duplicating the data layer adds complexity without proportional benefit since Aurora B/G already handles database switchover natively.

Approach B gives you full compute isolation (test EKS upgrades, new addons, Terraform module changes) while sharing the data layer (Aurora, ElastiCache, S3) that doesn't need blue/green. The spin-up cost (15 min for a new EKS cluster) is acceptable since you chose "spin-up on deploy."
