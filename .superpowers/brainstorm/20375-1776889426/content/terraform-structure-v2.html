How Blue/Green Toggle Works
# compute/main.tf (conceptual)

module "blue" {
  source     = "../modules/eks-stack"
  identifier = "${var.identifier}-blue"
  vpc_id     = var.vpc_id
  subnet_ids = var.private_subnet_ids
  # ... always exists
}

module "green" {
  source     = "../modules/eks-stack"
  count      = var.deploy_green ? 1 : 0
  identifier = "${var.identifier}-green"
  vpc_id     = var.vpc_id
  subnet_ids = var.private_subnet_ids
  # ... only exists during deployments
}

# routing/main.tf

resource "aws_route53_record" "app" {
  type = "CNAME"
  set_identifier = "blue"
  weighted_routing_policy { weight = var.blue_weight }   # 100 вҶ’ 0
  records = [module.blue.nlb_dns]
}

resource "aws_route53_record" "app_green" {
  count = var.deploy_green ? 1 : 0
  type = "CNAME"
  set_identifier = "green"
  weighted_routing_policy { weight = var.green_weight }  # 0 вҶ’ 100
  records = [module.green[0].nlb_dns]
}

Advantages of This Approach

  • Zero duplication вҖ” blue and green use the same module code, just different inputs
  • Single toggle вҖ” deploy_green = true spins up the entire green environment
  • Fewer Spacelift stacks вҖ” 4 per market instead of 6 (no separate blue/green stacks)
  • Clean teardown вҖ” set deploy_green = false and apply to remove green entirely
  • Role swap вҖ” after cutover, swap which instance is "blue" vs "green" via variable
