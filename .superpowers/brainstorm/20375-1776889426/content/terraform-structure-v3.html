What Changes vs Today
KEEPS:
  ✓ Two Spacelift stacks per market (physical + logical)
  ✓ Physical → Logical dependency chain
  ✓ Same Terragrunt directory structure
  ✓ Same module source references

ADDS:
  + deploy_green variable (bool, default false)
  + green_image_tag variable (when deploying)
  + blue_weight / green_weight variables (Route53)
  + Conditional second EKS cluster + NLB in physical
  + Conditional second helm release in logical
  + Dual provider configuration in logical (two K8s contexts)

CHANGES:
  ~ RDS MySQL → Aurora MySQL (shared, one-time migration)
  ~ EKS + NLB resources extracted into reusable local module
    or child module to avoid code duplication
  ~ DNS records change from simple CNAME to weighted CNAME set
Deployment Flow (Variables Only)
Steady state:
  deploy_green    = false
  blue_weight     = 100
  green_weight    = 0

Step 1 — Spin up green:
  deploy_green    = true
  green_image_tag = "abc123.6"        ← new version
  blue_weight     = 100               ← all traffic still on blue
  green_weight    = 0
  → spacectl: apply physical (creates green EKS + NLB)
  → spacectl: apply logical (deploys app to green cluster)

Step 2 — Canary (10%):
  blue_weight     = 90
  green_weight    = 10
  → spacectl: apply physical (updates Route53 weights)

Step 3 — Shift more (50/50):
  blue_weight     = 50
  green_weight    = 50

Step 4 — Full cutover:
  blue_weight     = 0
  green_weight    = 100

Step 5 — Promote green to blue:
  Swap: green becomes new blue, tear down old blue
  deploy_green    = false
  blue_image_tag  = "abc123.6"        ← was green's tag
  blue_weight     = 100
  green_weight    = 0

Why This Works

  • No new paradigm — physical creates infra, logical deploys apps. Same as today.
  • No new stacks — same 2 Spacelift stacks per market.
  • Toggle-driven — deploy_green=true/false is the only switch.
  • Internal modules — EKS+NLB extracted to a child module called twice. No code duplication.
  • Weight-based routing — Route53 weighted records in physical/dns.tf. Gradual or instant.

⚠️ One Complexity: Dual Kubernetes Providers

The logical layer needs to talk to two EKS clusters simultaneously (blue + green). Terraform 1.5.7 requires provider aliases for this:

provider "kubernetes" {
  alias = "blue"
  host  = data.aws_eks_cluster.blue.endpoint
  ...
}
provider "kubernetes" {
  alias = "green"
  host  = data.aws_eks_cluster.green[0].endpoint
  ...
}

# Each helm release targets a specific provider
helm_release "app_blue"  { provider = helm.blue  ... }
helm_release "app_green" { provider = helm.green ... }

This is the trickiest part — provider aliases with conditional resources. It works but requires careful structuring. The alternative is using two separate logical stacks (blue-logical + green-logical), but that breaks the two-stack paradigm you want to keep.
