Proposed Structure (Blue/Green)

│
├── shared/              ← Spacelift: usac-shared (NEW)

│                          Resources that outlive any single cluster
│
├── blue/
│   ├── physical/        ← Spacelift: usac-blue-physical
│   │   terragrunt.hcl      EKS cluster, NLB, Pod Identity,
│   │                       access entries, cert-manager PI
│   │                       References shared/ outputs (VPC, subnets)
│   │
│   └── logical/         ← Spacelift: usac-blue-logical
│       terragrunt.hcl      Helm chart, K8s resources, Vault auth,
│                           ESO, Gateway, TargetGroupBindings
│                           References blue/physical/ + shared/ outputs
│
├── green/               ← Only exists during deployments
│   ├── physical/        ← Spacelift: usac-green-physical
│   │   terragrunt.hcl      Same module, different cluster name
│   │
│   └── logical/         ← Spacelift: usac-green-logical
│       terragrunt.hcl      Same module, new image tag + config
│
└── routing/             ← Spacelift: usac-routing (NEW)
    terragrunt.hcl         Route53 weighted records only
                           Controls traffic split: blue=X, green=Y
                           Decoupled from both clusters

Complexity Assessment

Component Change Required Effort
CPI: New shared module Extract VPC, Aurora, ElastiCache, S3 from physical into new shared module Large
CPI: Slim down physical Physical becomes EKS-only: cluster, NLB, Pod Identity, access entries Medium
CPI: New routing module Small module: Route53 weighted records referencing blue/green NLB DNS Small
CPI: RDS → Aurora migration Replace aws_db_instance with Aurora cluster + instances Large (+ data migration)
infra-tf: Terragrunt restructure New directory layout: shared/, blue/, green/, routing/ per market Medium
Spacelift: Stack definitions Update locals.tf: 4 stacks per market instead of 2, new dependency graph Medium
Spacelift: Deploy orchestration Pipeline/hooks to orchestrate: spin up green → validate → shift traffic → teardown Large
State migration Move existing resources from physical state to shared state Medium (per market)

⚡ Overall Estimate

This is a multi-phase project spanning 4-6 weeks of focused infrastructure work:

  • Phase 1: RDS → Aurora migration (can be done independently, 1 week)
  • Phase 2: Extract shared module + state migration (1-2 weeks)
  • Phase 3: Blue/green physical/logical + routing modules (1 week)
  • Phase 4: Spacelift orchestration + testing (1-2 weeks)

Each phase is independently valuable and can be deployed to production incrementally.
